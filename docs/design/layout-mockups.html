The trade.

  • B gives the window obvious app-shape (two stacked rows = clear hierarchy) and keeps the HUD always-on so you never lose track of git/agent state.
  • B costs 50–60% more vertical chrome and ~20% horizontal real estate. At a small window (e.g. 100 cols) the HUD eats half the readable width.
  • D keeps the same identity (Basin mark + workspace + indicators) in one row, so the chrome reads as a tighter unit, not stacked.
  • D hides the HUD by default (⌘\\ toggles). You get full-width content when you don't need context; one keystroke when you do.
  • Both have the slim bottom bar — that's the single biggest "this is a stateful app" signal. Cheap and worth it.

B reads more like Linear/Xcode — stacked panels, always-on context.
D reads more like a modernized iTerm — one row of chrome, content takes the room.

For "premium without bloat," D is the right floor. B is the right ceiling if you want the HUD to be persistent. You could also start at D and offer B as a "show panel" option — ship both, one keybind toggles between them.

→ My read

The blocks are doing most of the work. Across A/B/C/D the chrome differs by tens of pixels but the terminal area looks identical and feels very different from a normal terminal because of the block treatment. That's the leverage.

  • A is the most premium-restrained. Risk: feels too light, like it's missing something.
  • B is more app-frame-y but spends ~50 extra vertical pixels on chrome. Probably too much.
  • C (floating tabs) feels softest / most Warp. Trades brand precision for warmth.
  • D is A + a bottom bar. Compromise. Best balance of "obviously an app" and "obviously not bloated."

If you want premium-restrained: D. If you want the absolute most minimal: A. Either way, blocks are the build.
